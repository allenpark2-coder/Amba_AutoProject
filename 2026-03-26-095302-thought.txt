     Cooper SDK 版本自動化管理計畫

     Context

     使用者定期從 Ambarella 收到 Cooper SDK tarball（基礎版本如 2.5，以及累進的
     patch 如 2.5.1、2.5.2），
     目前每次都需要手動解壓、手動建立 git repo、手動推送到 GitLab。
     目標是建立一套可重複執行的 Shell script 自動化流程，讓未來每次收到新 patch
     只需要執行一條指令。

     ---
     已確認的關鍵事實

     Tarball 結構（已驗證）

     - 基礎 SDK：cooper_linux_sdk_2.5_cv2x_cv5x_cv7x_20241219.tar.xz
       - 解壓後有容器目錄：cooper_linux_sdk_2.5_cv2x_cv5x_cv7x/
       - 內容：ambarella/、external/、doc/ 等
     - Patch：cooper_linux_2.5.2_cv2x_cv5x_cv7x_20250912.tar
       - 雙層結構：外層 .tar → 容器目錄 → 內層 .tar.bz2
       - 內層解壓後：cooper_patch_2.5.2_.../apply.sh + ambarella_*.sh
       - apply.sh 需要在 cooper_patch_.../ 目錄內執行，../ambarella/ 是 SDK 目錄

     GitLab 結構（來自 manifest XML）

     - Server：git@venus.vivotek.tw
     - 20 個 project repo，namespace：ad-odm-rdd3/codebase/sdk/amab_cooper_sdk/
     - Manifest repo：git@venus.vivotek.tw:odm-rd3/manifest/amba-cv5x.git
     - Branch 命名：COOPER_SDK-{VERSION}-{DATE}-{CHIPS}
     - Repo 由使用者手動在 GitLab 先建好；script 只做 push

     嵌套 repo 問題

     - ambarella/ 是父 repo，包含 18 個子目錄，每個都是獨立 repo
     - 需要在 git add 前為 ambarella/ 建立 .gitignore，排除所有子 repo 目錄名稱

     執行流程

     1. 讀取 project.json 取得 VERSION、TARBALL、CONTAINER、MANIFEST_XML、BRANCH
     2. 驗證 tarball 存在、manifest XML 存在
     3. 解壓 tarball 到 workspace/，再把容器目錄內容移出：
     tar xJf "$TARBALL" -C workspace/
     mv workspace/"$CONTAINER"/* workspace/
     rmdir workspace/"$CONTAINER"
     4. Pass 1 — 建立 .gitignore：
       - 對每個 path，檢查是否有其他 path 以 "$path/" 開頭
       - 若是父 repo，收集直接子目錄名稱 → 寫入 workspace/$path/.gitignore
     5. Pass 2 — git init 迴圈（對每個 path）：
     mkdir -p workspace/"$path"
     git init workspace/"$path"
     git -C workspace/"$path" add .
     git -C workspace/"$path" commit -m "Cooper SDK $VERSION" \
         || git -C workspace/"$path" commit --allow-empty -m "Cooper SDK 
     $VERSION"
     git -C workspace/"$path" checkout -b "$BRANCH"
     git -C workspace/"$path" remote add origin "git@venus.vivotek.tw:$name.git"
     git -C workspace/"$path" push -u origin "$BRANCH"
       - 已有 .git 的目錄跳過（冪等性保護）
     6. 更新 project.json：version status → "pushed"，current_version → VERSION
     7. 呼叫 ./push_manifest.sh "$MANIFEST_XML" "$BRANCH"

     執行流程

     1. 從 project.json 找到 type="patch" 且 status="pending" 的版本
     2. 確認 workspace/ambarella/ 存在（確保基礎版本已初始化）
     3. 解壓外層 tar，從中取出 inner .tar.bz2 路徑：
     mkdir -p workspace/_outer_$$
     tar xf "$OUTER_TAR" -C workspace/_outer_$$
     INNER_PATH=$(find workspace/_outer_$$ -name "$INNER_TARBALL" | head -1)
     4. 解壓 inner .tar.bz2 到 workspace/（讓 cooper_patch_.../ 和 ambarella/
     同層）：
     tar xjf "$INNER_PATH" -C workspace/
     5. 執行 apply.sh：
     PATCH_DIR=$(find workspace/ -maxdepth 1 -type d -name 'cooper_patch_*' |
     head -1)
     cd "$PATCH_DIR"
     ./apply.sh
     cd "$SCRIPT_DIR"
     6. 清除暫存目錄：刪除 workspace/_outer_$$ 和 workspace/cooper_patch_*/
     7. git 迴圈（讀取新 manifest XML，對每個 repo）：
     git -C workspace/"$path" checkout -b "$NEW_BRANCH"
     git -C workspace/"$path" add .
     # 如有變更才 commit，否則 --allow-empty
     git -C workspace/"$path" diff --cached --quiet \
         && git -C workspace/"$path" commit --allow-empty -m "Cooper SDK 
     $NEW_VERSION (no changes)" \
         || git -C workspace/"$path" commit -m "Cooper SDK $NEW_VERSION"
     git -C workspace/"$path" push origin "$NEW_BRANCH"
     8. 更新 project.json，呼叫 ./push_manifest.sh
